SSH keys and permissions: my quick checklist

When SSH key logins fail, the issue is usually not the key itself, but a small detail around permissions or configuration. Here is the checklist I run before I start "random troubleshooting".

The top 5 checks

  1. Am I using the correct user/host?
    Check the command: ssh user@host
  2. Is the public key on the server?
    It must be in ~/.ssh/authorized_keys.
  3. Permissions are strict
    Wrong permissions can make SSH ignore the key files.
  4. Server allows key auth
    Look for PubkeyAuthentication yes.
  5. Use verbose mode
    ssh -vvv shows what is happening.

Commands I copy into my write-up

# On the client
ssh -vvv -i ~/.ssh/<private_key> user@host

# On the server (as the target user)
ls -ld ~ ~/.ssh
ls -l ~/.ssh/authorized_keys

# Common permission fix (example)
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys

Placeholder: replace the key name and paths based on your setup.

Evidence I attach to my portfolio

  • A screenshot of a successful login
  • A screenshot of file permissions
  • The relevant sshd_config lines (or a snippet)
  • One ssh -vvv excerpt that shows the key was accepted
